# Contributing to dev-workflows

Thanks for considering a contribution. This handbook is meant to be a living document shaped by real experience.

## What Makes a Good Contribution

- **Battle-tested advice.** If you've used a practice on a real project and it worked (or failed instructively), that's valuable.
- **Concrete over abstract.** Show commands, config snippets, and examples. Avoid generic advice that could apply to anything.
- **Explain the tradeoff.** Every practice has a cost. Acknowledge what you're giving up, not just what you gain.

## How to Contribute

1. Fork the repo
2. Create a branch: `git checkout -b your-topic`
3. Make your changes — follow the existing document structure
4. Open a PR with a clear description of what you're adding and why

## Style Guide

- Use plain, direct language. Write like you're explaining to a colleague, not writing a textbook.
- Use second person ("you") not third person ("the developer").
- Include practical examples — command snippets, config files, directory trees.
- Mark opinionated recommendations with a note explaining that alternatives exist.
- Keep sections scannable: use headers, bullet points, and tables over long paragraphs.

## What We Don't Want

- Vendor-specific tutorials (use generic patterns, mention tools as examples)
- Advice that only applies at scale (100+ engineers) without noting that context
- Changes that remove the "why" behind a recommendation
